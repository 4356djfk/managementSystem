# 前端产品设计说明

## 1. 文档目标

本文档用于指导当前项目的前端建设，基于：

1. 已经完成的后端接口
2. 当前前端技术栈：`Vue 3 + Vite + Vue Router + Pinia`
3. 产品方向：**功能分布与界面风格尽量向 Microsoft Project 靠拢**

本文档重点回答四个问题：

1. 前端应该有哪些页面
2. 每个页面应该承载哪些功能点
3. 页面之间如何组织
4. 界面风格如何更接近 Microsoft Project

---

## 2. 产品定位

这个系统不建议做成普通“后台 CRUD 管理系统”。

更合适的产品形态应是：

1. 以“项目”为核心上下文
2. 以“任务 + 甘特图 + 项目工作区”为主工作界面
3. 风险、变更、成本、工时等围绕项目展开
4. 首页是“工作台”，不是展示型大屏

更接近的参考方向：

1. Microsoft Project
2. Project for the web
3. Microsoft Planner / Microsoft 365 工作台
4. Azure DevOps 中的工作项详情抽屉和列表布局

## 5. 核心设计原则

## 5.1 以项目为中心

项目详情不建议拆成很多彼此割裂的页面。  
建议做成“项目工作区”，进入单个项目后，通过左侧项目内导航切换不同模块。

## 5.2 以任务为主工作界面

任务页、甘特图页、里程碑页是最接近 Microsoft Project 的核心页面，应该优先设计。

## 5.3 高信息密度

不要走“卡片很大、留白很多、信息很少”的互联网运营风。  
应该偏企业项目管理工具风格：

1. 信息密度高
2. 表格能力强
3. 侧边属性面板多
4. 交互偏专业软件

## 5.4 页面不是 CRUD 堆叠

即使底层接口是 CRUD，前端也应尽量包装成工作流：

1. 列表 + 筛选
2. 详情抽屉
3. 视图切换
4. 工作区页签

## 8. 对标 Microsoft Project 的界面风格建议

## 8.1 视觉基调

建议采用：

1. 浅色企业风
2. 白底 + 浅灰背景
3. 蓝色主强调色
4. 小圆角
5. 弱阴影
6. 清晰分隔线

不建议：

1. 大面积渐变
2. 花哨插画风
3. 过度动画
4. 大块留白

## 8.2 重点界面形式

### 表格

应作为核心组件建设，要求：

1. 支持列筛选
2. 支持排序
3. 支持固定列
4. 支持行内编辑
5. 支持选中态

### 详情抽屉

建议大量使用右侧抽屉，而不是频繁页面跳转。

适用模块：

1. 任务
2. 风险
3. 变更
4. 成本
5. 需求

### 视图切换

任务页尤其要支持：

1. 网格
2. 看板
3. 甘特

### 页签工作区

适合模块：

1. 成本
2. 收尾
3. 报告

## 11. 当前结论

基于目前后端接口，前端已经可以支撑一套完整的项目管理产品界面建设。

如果目标是尽量接近 Microsoft Project，最关键的不是把页面做多，而是：

1. 先把项目工作区设计对
2. 先把任务网格和甘特图设计对
3. 再让风险、变更、成本、工时围绕项目工作区组织

也就是说，前端最优落地路径应该是：

1. 先搭壳子
2. 先做项目工作区
3. 先做任务与甘特图
4. 再逐步补全其他模块
